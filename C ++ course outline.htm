4.    Курс: Система контроля версий Git.

4.1             Общие сведения.

Задачи системы контроля версий:

1.   Сохранение промежуточных версий кода.

2.   Параллельная работа нескольких человек.

Сохранение промежуточных версий кода. Каждая версия имеет дату и время. Имеется ещё и название.

Можно попросить кого-либо добавить часть своей версии, которые впоследствии можно или принять, или отклонить. Каждая версия имеет термин commit – версия файла для фиксации его состояния.

Работа ведётся через командную строку. Одна команда глобальная, называется – git.

Системы контроля версий:

·        git status – показывает какие файлы готовы к добавлению в хранилище git;

·        git add<files> – добавляет созданные файлы в репозиторий;

·        git commit –m <message> – создаёт коммит и добавляет сообщение;

·        git log – показывает выполнение предыдущих команд и состояние добавленных файлов в репозиторий.

Три состояние файла:

                      git add                git commit

         Изменён -> Готов к коммиту -> Закоммичен

4.2             Установка git.

Скачать git с официального сайта http://git-scm.com, и запустить установку программы. При установке, необходимо выбрать редактор, используемый в git. Желательно, что бы это был редактор для разработки для лучшего взаимодействия с git.

Для работы с информацией, которую мы хотим сохранять, необходимо выполнить следующие шаги:

1.   Создать проект, который мы будем отслеживать (желательно в среде разработки, или папку с проектом, в котором мы будем работать с файлами.).

2.   Запустить командную оболочку bash.

3.   Используя команду cd [путь к папке], перейти в эту папку (имя текущей папки должно отображаться в строке команд).

4.   Выполнить команду git init для создания репозитория.

5.   Начать работу над проектом. При добавлении новых файлов в проект, необходимо отслеживать в bash, с помощью команды git status.

6.   При появлении новых или изменённых файлов, необходимо их добавить в индекс. Для этого используется команда
 git add . (обратите внимание на символ «.» через пробел после add, это обязательно, такой синтаксис команды).

7.   Проверяем состояние индексов с помощью команды git status. Изменения, включённые в индекс выделены зелёным цветом (см. рисунок 4.1).

8.   Добавить изменения в репозиторий с помощью команды git commit –m ‘initial commit’.

9.   Проверить состояние репозитория с помощью команд git status и git log.

4.3             Индекс и частичные коммиты.

В этом уроке будут подробно изучены команды:

·        git add

·        git status

·        git rm

·        git mv

Изменения можно отслеживать и выполнять команду git add можно и в среде разработки.

Команда git status состоит из секций:

1.   отображает файлы, включенные в индекс (Changes to be committed). Эти файлы подсвечиваются зелёным цветом.

2.   отображает файлы, отсутствующие в индексе (Changes not staged for commit). Такие файлы подсвечиваются красным цветом.

3.   файлы, не зарегистрированные в системе git (Untracked files). Подсвечиваются красным цветом.

Команда git add позволяет указать только те файлы, которые надо добавить в репозиторий, например – git add styles.css script.js добавит в индекс только указанные файлы.

Команда git add позволяет добавлять в индекс не весь изменённый файл, а только его часть. Синтаксис команды: git add –p <имя файла>. После этого будут предложены варианты добавления фрагмента (см. ниже), если выбрать параметр s, будет добавлен только часть файла, разбитого на фрагменты.

Опции команды git add –p <имя файла>:

Добавить в индекс эту часть [y,n,a,d,/,j,J,g,e,?]? ? y - добавить в индекс эту часть

n - не добавлять в индекс эту часть

a - добавить в индекс эту и все оставшиеся в этом файле части

d - не добавлять в индекс эту и все оставшиеся в этом файле части

g - перейти к некоторой части файла (g - показывает список частей и затем выполняет переход, g<N> - перейти к части N)

/ - найти часть, соответствующую регулярному выражению

j - отложить принятие решения по этой части, перейти к следующей части, решение по которой не принято

J - отложить принятие решения по этой части, перейти к следующей части

k - отложить принятие решения по этой части, перейти к предыдущей части, решение по которой не принято

K - отложить принятие решения по этой части, перейти к предыдущей части

s - разбить текущую часть на части меньшего размера

e - вручную отредактировать текущую часть

? - отобразить помощь

Для того, чтобы некоторые файлы не отслеживались системой git и не добавлялись в репозиторий, необходимо создать файл .gitignor, и добавить его в репозиторий. В указанном файле должны указываться файлы и папки, которые не надо добавлять в репозиторий. Если файл ранее отслеживался системой git, а затем был добавлен в список игнорируемых, он все равно будет отслеживаться. Чтобы убрать его с контроля системой git, необходимо выполнить команду git rm –cached <имя файла>. Указанный файл будет удалён из системы версий. ВНИМАНИЕ! Если выполнить команду git rm без ключа –cached, файл будет удалён и из системы git, и из файловой системы.

Команда git mv позволяет переименовать файл. При этом переименованный файл будет обновлён в индексе автоматически. Останется только сделать коммит для закрепления изменений.

Хотя системы разработки и позволяют отслеживать изменения в контроле версий, но оболочка bash является более удобным инструментом с полным набором функций управления git.

4.4             Сравнение версий.

Команда git log позволяет просмотреть все коммиты, перемещение по коммитам с помощью стрелки вниз/вверх, выход из просмотра – клавиша q. Есть ещё команда git show, которая показывает последний коммит. Здесь же показываются последние изменения, которые были сделаны в рамках этого коммита.

В этом уроке будет показано как сравнивать разные версии и как просматривать последние изменения. Это можно делать со среды разработки, поддерживающих git.

Для просмотра из командной строки (bash), используется команда git log для просмотра новых коммитов. Команда git diff <первый коммит> <второй коммит> показывает изменения, которые были внесены. В этом режиме показываются удалённые строки (знак “-“ слева) и добавленные строки (знак “+” слева). В качестве указателей на коммиты использовались их хеш. В качестве указателей можно использовать макрос HEAD с указателем номера последнего коммита через знак “~”. Например, git diff HEAD~2 будет показана информация, как и в варианте команды с использованием хеша для последнего коммита. Вариант команды git diff HEAD~2 HEAD~1 будут показаны изменения относительно последнего и предпоследнего коммита. Команда git diff ––name only <хеш коммита ниже по ветке> покажет все фалы, которые были изменены/добавлены/удалены за время относительно указанного коммита.

Использование среды разработки для просмотра изменений более наглядный, так как показывает не только добавленные или удаленные строки, но строки в которых были частичные изменения. Кроме этого можно просмотреть цепочку коммитов и информацию по ним.

В данном уроке были рассмотрены команды:

·        git show

·        git diff <commit1> <commit2>

·        git diff <commit>

·        git diff <file> / git diff HEAD

·        git diff ––name-only <params>

4.5             Отмена изменений и откат версий

В данном уроке будут рассмотрены способы отмены изменений. Если по каким-то причинам необходимо отменить последние изменения можно использовать команды:

·        git checkout --<имя файла> – для отмены изменений, которые были внесены, но не добавлены в индекс. Если выполнить команду git status, такие файлы будут подсвечены красным цветом с пометкой modified.

·        git reset HEAD <имя файла> – для отмены изменений и добавления изменённого файла в индекс (сформирован хеш) командой git add. В этом случае файл будет удалён из индекса и возвращён в область до добавления изменений. Команда git status покажет файл с пометкой modified с выделением красным цветом.

·        git commit ––amend – изменить предыдущую фиксацию если изменённый файл бал добавлен в коммит (выполнена команда git commit –m ‘комментарий’). Если использовать эту команду сразу после коммита, то эта команда открывает окно редактора, и предлагает изменить комментарии к коммиту. После изменения комментария и закрытия редактора в репозитории будет изменён только комментарий, ни сам файл, ни индекс изменены не будут. Если в файл были внесены изменения и после этого выполнить эту команду, то опять будет предложено изменить только комментарий, при этом может изменится индекс коммита. Для того чтобы заменить в коммите сам файл после изменения, необходимо выполнить следующие команды:

o   git commit -m 'initial commit'

o   git add forgotten_file

o   git commit –amend

В итоге получится единый коммит — второй коммит заменит результаты первого.

 

 

 

 

 

 

 

 

 

 

 

 

 

 

 

 

Вы добавили еще один репозиторий git в свой текущий репозиторий. Клоны внешнего репозитория не будут содержать содержимое встроенного репозитория и не будут знать, как его получить.

Если вы хотели добавить подмодуль, используйте:

         git submodule add <url> VD

Если вы добавили этот путь по ошибке, вы можете удалить его из индекса с помощью:

         git rm --cached VD

См. «git help submodule» для получения дополнительной информации.
